Contest: Soybean Grand Challenge
Problem: PedigreeEliteClassifier

Problem Statement

    

Prizes

This contest has a $15,000 total prize purse. $3,000 of them are awarded as two sets of milestone prizes.

The first set of milestone prizes is awarded as follows: we take all solutions submitted no later than 1 week after the start of the match (1 PM, April 22; only last submission from each member) and evaluate them on all system test cases. The 3 highest scored submissions are milestone winners.

The second set of milestone prizes is awarded in the same fashion, but the deadline is 2 weeks after the start of the match (1 PM, April 29). Both sets of milestone prizes use the following prize distribution:

1st place -- $750
2nd place -- $500
3rd place -- $250

Since milestone results (and even the handles of milestone winners) reveal some information about system test cases, we will announce them only after the end of the match.

The final prize of $12,000 are awarded to 5 highest scoring submissions as follows:

1st place -- $6,000
2nd place -- $2,400
3rd place -- $1,800
4th place -- $1,200
5th place -- $600

In case of any ties, the corresponding prize money are distributed equally among all tied competitors. The date/time of submission does not matter.

Overview

Every year, plant breeders try to create new varieties that will outperform those which are currently available on the market. New varieties are created by cross-pollination. In this competition you will try to predict the quality of such varieties based solely on their pedigree (the choice of parents).

Past solutions

You can reuse code and/or ideas from best 5 submissions from each of the past soy contest. You can download all of them here along with the descriptions provided by their authors. Since this contest is a mix between PedigreeYieldPredictor and AdvancedEliteClassifier, solutions from those two contests should be much more useful than from others.

Experiment

Experiment is the process of planting the specified set of varieties and collecting all of the relevant information during the harvesting of the plant. Such information includes grain yield, date of maturity.

Each experiment consists of some fixed number of different varieties. Since there is much variability involved in the whole process and varieties usually perform differently in different environments, each experiment is run independently in several trials spread among different locations.

For further information about whole process you can refer to Knowledge Base.

Data

This competition uses historical data obtained from many years of soy breeding. It consists of several thousands of experiments.

While we provide some training data that can be used for offline analysis, in this competition you are required to train your solution only using the data passed to your methods as arguments. This means that your solution should not contain any fragments of training data or anything else that was derived from it.

Three files are available for you: TrainingExperiments.csv, TrainingMaterial.csv and Location.csv to download. Both TrainingExperiments and TrainingMaterial files contain a subset of the available data that will be given to you in the train method. To be more precise the data up to 2002 is complete, and from 2003 onwards it represents only a subset of actual data. Location file contains all the available information.

Experiments Data

Each line contains information about performance of a single variety within a single experiment for a single rep.

Format:

  • ExperimentID - Each experiment has a different ID.
  • Stage - Stage number of experiment. This ranges from 3 to 7 inclusive.
  • YEAR - Year in which the experiment was done.
  • Location - You can refer to Locations.csv to obtain additional information about each location. Please note that such information is not available for all locations.
  • Rep - Number of the repetition of one trial.
  • MaterialID - ID of the variety within experiment.
  • EntryID - ID of variety within experiment. This is usually redundant, but it's provided for completeness.
  • PlantDate - The date when the variety was planted. It is given in form "YYYY-MM-DD", for example, "2000-07-15". NULL if it's not available.
  • MaturityDate - The date when the variety was declared as mature. Same for as PlantDate. NULL if it's not available.
  • HarvestDate - The date when the variety was harvested. Same for as PlantDate. NULL if it's not available.
  • Yield - Yield of a variety given in bushels per acre. NULL if it's not available.
  • MN - Maturity number of a variety. By definition, this should be equal to the day difference between PlantDate and MaturityDate. NULL if it's not available.
  • PD_CR - Pod Color trait. Between 1 and 9. NULL if it's not available.
  • PB_CR - Pubescence Color trait. Between 1 and 9. NULL if it's not available.
  • FL_CR - Flower Color trait. Between 1 and 9. NULL if it's not available.
  • EMRGR - Emergence trait. Between 1 and 9. NULL if it's not available.
  • PLHTN - Plant height trait (in centimeters). NULL if it's not available.
  • HLDGR - Harvest Lodging trait. Between 1 and 9. NULL if it's not available.
  • IsCheck - 1 if a variety was used as check, 0 otherwise

To reduce the size of the data, all NULL values were replaced with an empty string. For example "NULL,354,NULL,13" will be converted to this ",354,,13".

Materials Data

Each line contains information about one variety.

Format:

  • MaterialID - ID of the variety. This is the same ID that is used in experiments data.
  • Pedigree - Pedigree information available for the variety. You can find more information here. NULL if it's not available.
  • Type - Type of the variety. It's equal to NULL or one of the following 3 values : conv (conventional), RR1 (Roundup Ready) or RR2Y (Roundup Ready To Yield).
  • RM - Relative maturity of a variety. Zone 00 is represented with the [-1, 0) range, and zone 000 is represented with the [-2, -1) range. NULL if it's not available.
  • Year - Year in which the variety was used for the very first time.
  • IsAdv - 1 if a variety was declared as advanced, 0 otherwise.
  • IsElite - 1 if a variety was declared as elite, 0 otherwise.

To reduce the size of the data, all NULL values were replaced with an empty string. For example "NULL,354,NULL,13" will be converted to this ",354,,13".

Locations Data

Each line in Locations.csv contains information about one geographical location. Each (LOCCD, Year) point to one physical location - this means that one location code (LOCCD) can point to different physical locations through out different years. You don't get information of the matching location - instead you're given the properties describing the maturity in given location.

Format:

  • LOCCD - Location code.
  • Year - Year.
  • Maturity Zone - "Row" position on the map.
  • Maturity Band - "Column" position on the map.
  • Maturity Sub-Zone - There are only 3 different values : Early, Mid & Late.

Implementation, Testing

Your task is to write a program that will implement three methods: train, predictAdv and predictElite. The train method will be called first, and this is the place where we will give you all data that you can use for training your algorithm. Next predictAdv and predictElite are called (in that order) - in each of those, you should return an array of integers, where each value represents the MaterialID. You should rank the varieties given to you in the pedigreeData argument, based on the probability that given variety is advanced or elite (IsAdv and IsElite fields respectively).

In each test, all the varieties with unknown qualities, come from a single year. The data you'll receive will contain everything up to that year (but not including it). For example, if the test relies on data from year 2005, then experimentData and materialData will contain all experiment and material data up to year 2004. locationData is always identical to Location.csv file. pedigreeData contains the information about the new varieties, where each element is of a following format : MaterialID,Pedigree,Type (the meaning of those fields is explained in the Material Data section).

Return value from train method is ignored and you can return any value you would like.

Your return value from both predictAdv and predictElite should contain the same number of elements as given pedigreeData argument. And each element should be equal to one of the materialIDs.

Scoring

Both predict methods are scored independently and each of them constitutes 50% of your score for a given test. The scoring for methods predictAdv and predictElite works as follows. First we split the varieties into 3 groups, based on their Type: conv, RR1 or RR2Y. Again, each group is scored independently and it's score is based on the Average Precision metric. But this time each score has weight proportional to the number of advanced/elites in each group. The weights are set in such a way, that the maximum possible score for a test case is 1000000. The following pseudo code will be used to calculate Average Precision for 1 type:

score := 0.0
total := 0
correct := 0
for i := 0 .. size(A) - 1
    if (type(A[i]) != current_type) continue
    total := total + 1
    if (is_correct(A[i])) {
        correct := correct + 1
        score := score + (correct / total)
    }
score := score / correct

where A represent your return value, and is_correct returns true if variety is considered advanced/elite, false otherwise.

If your solution exceeds time or memory limit, crashes or returns invalid result, your score for this test case is 0. Your result is considered valid if it contains MaterialID of all of the given varieties and each MaterialID occurs only once.

The final score is a sum of scores obtained on each of the tests. This is the score you will on the leaderboard.

Special rules

  • You can include the open source code in your submission. Open Source libraries must be issued under the BSD license only. No other open source license can be accepted (e.g. GPL, GPL2, etc).
  • The solutions can be implemented on C++, Java, C# and VB. Python is not available for this match.
  • We reserve the right to not release the data used in provisional and/or final testing.
  • In order to receive the prize money, you will need to fully document how your algorithm works. Note that this description should NOT be submitted anywhere during the coding phase. Instead, if you win a prize, a TopCoder representative will contact you directly in order to collect these data. The description must be submitted within 7 days after the contest results are published.
  • You may not use any external (outside of this competition) source of data - this also means that you're not allowed to use any other data from other contests from Soybean MM series.
  • While we provide some training data that can be used for offline analysis, in this competition you are required to train your solution only using the data passed to your methods as arguments. This means that your solution should not contain any fragments of training data or anything else that was derived from it.

Notes

-Time limit is 10 minutes per test case and memory limit is 1024MB. The time limit is shared among all of the 3 methods.
-There is no explicit code size limit. The implicit source code size limit is around 1 MB (it is not advisable to submit codes of size close to that or larger). Once your code is compiled, the binary size should not exceed 1 MB.
-The compilation time limit is 30 seconds. You can find information about compilers that we use and compilation options here.
-There are 2 example tests. First test contain data from year 2001 and the other contain data from year 2002.
-There are 8 provisional and 8 final tests. The tests contain the data from years 2003 - 2010, 1 test per year. Provisional tests use approximately 25% of varieties, and final tests use the ones that were not used in provisional tests.
